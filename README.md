# 陣列兩兩 XOR 演化模擬器 (Pairwise XOR Evolution Simulator)

## 專案概述
本專案旨在模擬一個整數陣列的「兩兩互斥或 (Pairwise XOR) 演化過程」。
給定一個初始陣列，在每一個演化步驟中，程式會將陣列中**所有可能的兩兩組合**進行 XOR 運算，產生一個全新的陣列，並反覆迭代。

### 核心挑戰：組合爆炸 (Combinatorial Explosion)
陣列的長度 $N$ 增長公式為 $N_{next} = \frac{N(N-1)}{2}$。
長度會以平方級的速度暴增，導致傳統的 $O(N^2)$ 演算法在幾步之內就會面臨記憶體溢位 (OOM) 或運算卡死。本專案完整紀錄了從「硬體壓榨」到「數學降維打擊」，再到「多核心並行擴展」的極致優化歷程。

## 終止條件與宏觀物理意義

為了精準捕捉演化過程中的狀態轉換，本引擎定義了以下終止條件與物理觀測指標：

### 1. 基礎狀態收斂
* **收斂 (Converged)：** 陣列長度縮減為 1。
* **常數循環 (Constant Value Loop)：** 陣列內所有元素皆相同，系統處於高度均勻狀態。
* **空陣列防護 (Empty Array)：** 處理無效輸入的基礎防線。
* **長度超載 (Too Large)：** 陣列長度超過系統負荷或人為設定上限。

### 2. 進階宏觀觀測指標 (Engine Layer)
* **全零滅絕狀態 (Zero Extinction)：** 因為 $x \oplus x = 0$，若陣列演化到全部元素皆為 `0`，在數學上即進入「吸收態 (Absorbing State)」。系統失去所有資訊與能量，程式直接宣告演化終止。
* **最大位元勢能崩潰 (MSB Energy Collapse)：** XOR 永遠不會產生比輸入最高位元 (MSB) 更大的數字。若觀測到陣列整體的 MSB 發生斷崖式下降，代表系統複雜度與「能量」正在不可逆地坍縮 (Subspace Confinement)。
* **長度宿命論與 $N=3$ 奇異點：**
  長度序列並無週期性，而是嚴格遞增發散。唯一的例外是 **$N=3$**，此時 $3 \times 2 / 2 = 3$，陣列將永遠被困在長度為 3 的維度中，形成數學不動點。
* **循環偵測 (Loop Detected)：** 精準判斷系統頻率分佈是否陷入封閉迴圈，並計算其震盪週期。

## 演算法演進史 (Version History)

本專案經歷了四個層次的典範轉移：**純工程實現 $\to$ 硬體壓榨 $\to$ 數學降維 $\to$ 企業級架構與並行化**。

| 版本 | 核心策略 / 優化重點 | 痛點與突破 |
| :--- | :--- | :--- |
| **v1-v2** | **[純工程實現]** 雙層迴圈暴力配對，基礎 Hash 狀態記錄。 | 純 Python 處理 $O(N^2)$，長度過千即卡死，記憶體消耗極大。 |
| **v3-v6** | **[硬體壓榨]** 引入 PyTorch，利用廣播矩陣 (`t.view(-1, 1) ^ t`) 轉移至 GPU，實作分塊計算 (Chunked) 避免 VRAM 溢位。 | 將硬體算力壓榨至極致，但無法改變 $O(N^2)$ 複雜度本質，遇百萬級長度仍極度耗時。 |
| **v7-v8** | **[數學降維]** 徹底放棄記錄元素，改用「頻率直方圖」。引入 **快速華許轉換 (FWHT)** 進行 XOR 卷積。 | **[突破]** 記憶體降為 $O(1)$，時間複雜度降為 $O(M \log M)$。但早期實作存在 Hash 不穩定與防護漏洞等技術債。 |
| **v10** | **[企業級架構重構]** 採用策略模式 (Strategy Pattern)，將觀測引擎與演算法解耦。修復所有邊界邏輯與步數語義錯誤。 | **[突破]** 引入 $O(1)$ 滾動歷史查詢，並確保 Python BigInt (無限大整數) 的 Hash 絕對安全，支援極端壓力測試。 |
| **v10-Parallel** | **[多進程並行]** 引入 `multiprocessing` (ProcessPoolExecutor)。 | **[突破]** 利用尷尬並行 (Embarrassingly Parallel) 特性，無視 GIL 限制，榨乾多核心 CPU 算力進行批次測試。 |

## 核心架構設計 (v10)

最終版本採用了嚴謹的物件導向設計，確保高內聚與低耦合：

### 1. 演化引擎 (`EvolutionEngine`)
* **單一職責 (SRP)：** 專注於生命週期管理、終止條件檢查、MSB 勢能觀測以及歷史軌跡比對，不干涉數學計算。
* **安全與效能：**
  * 使用 `hashlib.blake2b` 確保跨程序的 Hash 穩定性與防碰撞。
  * 實作尾端除零 (Trailing Zero Truncation) 標準化狀態，縮減雜湊開銷。
  * 全面支援 **Python 無限大整數 (BigInt)**，避免底層 C 語言 64-bit 結構溢位崩潰。
  * 透過 `deque` 與 `dict` 實現 FIFO 的 **$O(1)$ 滾動歷史淘汰機制**，消除線性掃描瓶頸。

### 2. 演化策略 (`fwht_strategy`)
* 負責執行具體的 XOR 頻率計算。
* 採用 **Fast Walsh-Hadamard Transform (FWHT)**，將原本 $O(N^2)$ 的組合問題轉化為 $O(M \log M)$ 的頻率空間自我卷積（$M$ 為陣列中最大數字所佔的二進位空間大小）。

### 3. 並行調度器 (`Parallel Dispatcher`)
* 透過進程池 (Process Pool) 隔離記憶體狀態，確保高併發測試時的線程安全。
* 非同步運算與同步渲染：後台無序運算，前台依序渲染報表，確保 CLI 輸出的可讀性。

## 效能展示 (Performance Benchmark)

在 FWHT 演算法與並行架構的加持下，程式能在數毫秒內完成極端壓力測試：
* **輸入：** `list(range(100))` (包含 0~99 共 100 個不重複數字的百連殺測試)
* **演化表現：** 在短短 10 步之內，陣列虛擬長度突破 **$10^{1737}$**（一個 1738 位數的超級大數）。
* **資源消耗：** 記憶體佔用全程保持常數，單一 CPU 核心耗時僅需 **~5 毫秒 (ms)**。
