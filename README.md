# 陣列兩兩 XOR 演化模擬：程式碼版本演進與終止條件分析

## 一、 程式碼核心目的
本系列程式碼旨在模擬一個陣列的「兩兩互斥或 (Pairwise XOR) 演化過程」。
給定一個初始整數陣列，在每一個演化步驟中，程式會將陣列中**所有可能的兩兩組合**進行 XOR 運算，產生一個全新的陣列，並將此過程反覆迭代。

此問題最大的挑戰在於**組合爆炸 (Combinatorial Explosion)**：陣列長度 $N$ 會以 $O(N^2)$ 的速度暴增。因此，程式碼的演進史，就是一段與「執行時間」和「記憶體溢位 (OOM)」對抗的優化過程。

## 二、 現有與更多潛在的終止條件

為了避免程式陷入無限迴圈或耗盡系統資源，設定精準的「終止條件」至關重要。

* **收斂 (Converged)：** 陣列長度縮減為 1。
* **常數循環 (Constant Value Loop)：** 陣列內所有元素皆相同（例如 `[2, 2, 2, 2]`），下一步必然產生全 0。
* **偵測到循環 (Loop Detected)：** 利用 Hash 或精確比對，發現陣列狀態（或排序後的狀態）與歷史軌跡重複。
* **長度超載 (Too Large / Memory Limit)：** 陣列長度或預估記憶體超過系統負荷。
* **達到最大步數 (Max Steps Reached)：** 強制停止的保險機制。
* **穩定規律 (Stable Pattern - *test6 獨有*)：** 連續多步的「相異數字個數」、「最大值」、「最小值」皆未改變，視為宏觀穩定。
* **「全零」滅絕狀態 (Zero Extinction)：** 因為 $x \oplus x = 0$，如果陣列演化到某一步變成全部都是 `0`，這在數學上是一個「吸收態 (Absorbing State)」。下一步配對依然是全 0，可以立刻宣告系統「死亡/歸零」並終止。
* **長度序列週期性 (Length Sequence Periodicity)：**
  當陣列極大時，記錄完整陣列的 Hash 仍很耗資源。我們可以只追蹤「每一步的陣列長度數值」。如果長度變化呈現 $L_1 \to L_2 \to L_3 \to L_1$ 的週期，即使內部元素不同，系統結構也已陷入發散迴圈，可提早終止。
* **最大位元勢能崩潰 (MSB Energy Collapse)：**
  XOR 運算有一個特性：**永遠不會產生比輸入最高位元 (MSB) 更大的數字**。如果監控到陣列的整體最大值在某幾步內發生斷崖式下降（例如位元數從 8-bit 掉到 2-bit 再也上不去），代表系統能量正在收斂，可作為一種子空間降維的判定條件。
* **空陣列防護 (Empty Array)：**
  當輸入陣列長度為 0 或在某些過濾變體中長度歸零時，應立即終止。

## 三、 各版本程式碼差異與演進總覽

從 `test1.py` 到最終的 `test7_fwht.py`（降維打擊版），展現了從純邏輯實現到極致效能優化的三個階段：**純工程實現 $\to$ 硬體壓榨 $\to$ 數學降維**。

| 版本 | 核心策略 / 優化重點 | 痛點與極限 |
| :--- | :--- | :--- |
| **test1.py** <br>*(直觀原型版)* | 使用 Python 內建 `itertools.combinations` 產生配對並計算 XOR。用 `tuple` 記錄歷史以防無窮迴圈。 | 雙層迴圈極慢，長度過千就會卡死，`tuple` 極度消耗記憶體。 |
| **test2.py** <br>*(雜湊優化版)* | 引入 `hashlib.md5` 將陣列轉為 Hash 字串記錄歷史狀態，降低記憶體佔用。區分「精確比對」與「無序多重集合比對」。 | 核心算力未變，純 Python 處理 $O(N^2)$ 依然是效能瓶頸。 |
| **test3.py** <br>*(PyTorch 向量版)* | 引入 PyTorch，利用廣播矩陣 (`t.view(-1, 1) ^ t`) 與 `triu_indices`，將運算轉移至 GPU/CPU 向量指令。 | 速度躍升百倍，但一次性產生 $N \times N$ 矩陣會瞬間撐爆 GPU 記憶體 (VRAM)。 |
| **test4.py** <br>*(進度追蹤版)* | 在前幾版的基礎上，套用 `tqdm` 模組封裝迴圈。 | 主要為開發者體驗 (UX) 優化，方便觀察每一步陣列長度的「爆炸性」增長。 |
| **test5.py** <br>*(記憶體極限版)* | 解決 VRAM 暴增問題。實作「分塊計算 (Chunked)」與「生成器流式處理 (Generator)」，並控制 Hash 歷史的數量上限。 | 把硬體壓榨到極致，但仍無法改變陣列長度呈平方增長的物理事實。 |
| **test6.py** <br>*(生產環境工程版)*| 高成熟度模組。加入型別標註、動態偵測硬體記憶體決定分塊大小，並引入「穩定規律 (Stable Pattern)」啟發式提早終止條件。 | 工程層面的最優解，但遇上長度超過百萬的陣列仍需漫長運算。 |
| **test7.py** <br>*(FWHT 降維打擊版)*| **[範式轉移]** 徹底放棄記錄陣列元素。改用「頻率直方圖 (Frequency Map)」記錄每個數字出現次數，並利用 **快速華許轉換 (FWHT)** 進行卷積。 | **終極版本**。記憶體消耗變為常數 $O(1)$，時間複雜度降為 $O(M \log M)$。可輕鬆計算長度突破 $10^{1000}$ 的陣列，只需普通 CPU 即可秒殺 GPU。 |
